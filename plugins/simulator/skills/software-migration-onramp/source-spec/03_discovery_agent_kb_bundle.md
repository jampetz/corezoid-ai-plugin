# Discovery Agent — Knowledge Base Bundle v1.1

**Документ для prompt engineering / ML команды.**
**Дата:** 17 мая 2026
**Связанные документы:** Discovery Agent Spec v1.3, Master Spec v1.4.
**Изменение от v1.0:** Добавлен Onramp Process pack как RAG-источник, обновлены Quality Gates с учётом roadmap generation в Этапе 5.

Этот документ описывает, **что именно** должна знать AI-актор `DiscoveryAgent`, как это организовано в его контексте и как измерять качество его работы.

---

## 0. Зачем этот документ

Discovery-агент работает на LLM (Claude). Его поведение определяется четырьмя слоями знаний:

1. **System prompt** — фиксированные принципы, роль, ограничения. Лежит в спецификации.
2. **Knowledge base (RAG)** — отраслевая экспертиза, эталонные документы, прецеденты. Подгружается по релевантности в каждый turn.
3. **Few-shot examples** — пары «входной диалог → правильное поведение/артефакт».
4. **Live context** — текущая фаза диалога, состояние Lead-актора, история сообщений с клиентом.

Этот документ фиксирует слои 2-3 для первой версии агента под Furniture Retail.

### 1.2a. Onramp Process pack (мета-Industry-Pack) — новое в v1.1

Onramp Process pack — это шаблонный actor graph **самого процесса миграции**, который DiscoveryAgent клонирует и кастомизирует для каждого клиента через tool `generate_roadmap_graph`. Содержит 4 default roadmap templates (L1, L2, L3, L4) с Phase actors, dependencies, target dates.

**Структура pack'а:**
- Templates по 4 трекам (L1 Greenfield, L2 Single replacement, L3 Multi-system, L4 Enterprise)
- Каждый template = directed acyclic graph из Phase actors
- Каждый Phase actor имеет identity (phase_name, owner_agent, depends_on, target_date_weight, signoffs_required), state, accumulator accounts
- Deployment-specific modifiers (например, On-prem track добавляет "On-prem setup" phase)

**Для DiscoveryAgent — что использовать в Этапе 5:**
- При классификации трека (`classify_track`) → определяется какой template L1-L4 клонировать
- При `generate_roadmap_graph` → template клонируется, target dates распределяются от today до target_cutover_date, custom phases добавляются если клиент упомянул специфику
- Output показывается клиенту как graph/Gantt/calendar

**Сигналы клиента, которые означают custom phases:**
- «нужно одобрение совета директоров» → custom Phase "Board approval" перед Cutover
- «у нас регуляторная проверка в Q3» → custom Phase "Regulatory review window" с blocked status
- «дайте нам летние каникулы август-сентябрь» → built-in PAUSE intent
- «работаем фазами по 1С-модулям» → staged cutover с Phase per module

**Owner:** Solution Architects Lead. Pack эволюционирует на основе реальных миграций — какие custom phases добавлялись чаще всего, какие сроки реальны.

### 1.4. Каталог конкурентов и позиционирование

Краткие карточки по основным альтернативам, с которыми клиенты сравнивают:

## 2. Few-shot example — Golden House

Этот пример вкладывается в system prompt как «вот как выглядит идеальный диалог».

### Вход — транскрипт звонка Никиты с Надеждой (саммари)

(сокращённая версия документа, который Александр загрузил первым в треде)

> Надежда Сокол, представитель сети мебельных салонов Golden House в Днепре. Текущая система — самописка. Цель — заменить на украинский программный продукт до 2027-2028 года из-за введения НДС и требований ФОП. Российский 1С неприемлем, БАС не подходит. Потребности: замена учёта, управленческий учёт для розницы, поддержка полной цепочки сделки от заявки до отгрузки и финрезультата, работа менеджеров в салонах, склад + логистика + поставщики, готовый отраслевой продукт (она ожидала TorgSoft), возможность миграции на «официал» при введении НДС, фильтрация заказов по поставщикам и стадиям.
>
> Бизнес-флоу: клиент в салоне → менеджер оформляет «Замовлення клієнта» → предоплата + ПРРО → заказ поставщику (≈30 дней изготовления) → доплата + ПРРО перед готовностью → транспорт → склад → отгрузка клиенту → финальные документы → закрытие.
>
> Параллельно работают склад, отдел поставщиков, логистика. Один заказ может состоять из нескольких подзаказов разным поставщикам.

### Ожидаемый выход — что должен собрать агент

**Lead-актор после Discovery с Надеждой должен иметь заполненными:**

*Идентификационные Accounts:*
- `company_name = "Golden House"`
- `industry = "furniture_retail"`
- `outlets_count = N` (уточнить)
- `employees_count = N`
- `current_system = "самописка"`
- `migration_trigger = "ндс_фоп_2027"`
- `target_deadline = "2027-2028"`

*Case Accounts (заполнены описаниями):*
- `case_1_retail_flow` = «клиент в салоне → менеджер оформляет Замовлення клієнта → актор Order связан с Customer, Showroom, Employee»
- `case_2_prepayment_flow` = «предоплата через ПРРО прямо в салоне, провайдер уточнить»
- `case_3_vendor_flow` = «email/телефон поставщику после предоплаты, поставщик присылает счёт со своей ценой и сроком ~30 дней, цена розница/опт хранится в системе»
- `case_4_lead_time_flow` = «менеджер вручную отслеживает дату готовности, уведомления не автоматические»
- `case_5_final_payment_flow` = «доплата перед прибытием товара, тоже через ПРРО»
- `case_6_logistics_flow` = «транспорт наёмный, везут от поставщика на свой склад, потом клиенту»
- `case_7_closing_flow` = «подписание финальных документов закрывает цепочку»
- `case_8_returns_flow` = (уточнить у Надежды — сейчас неизвестно)

*Volumes & Infra:*
- `orders_per_month`, `contractors_count`, `sku_count` — все уточнить
- `prro_provider` — уточнить
- `banks[]` — уточнить
- `db_export_available` — спросить

*Specifics:*
- `must_keep` — «фильтрация заказов по поставщикам и стадиям» (явно озвучено)
- `top_pain` — «текущая система не выдержит требований НДС/ФОП», «нужна полная цепочка в одной системе»
- `decision_maker` — уточнить (Надежда сама или собственник?)
- `escalation_signals[]` — пусто (нет красных флагов)

*Computed:*
- `industry_pack_match_pct` ≈ 90% (Furniture Retail pack покрывает все 8 кейсов)
- `predicted_track = "standard"` (multi-salon, длинная цепочка, но <30 сотрудников вероятно — на грани с Express)

**Brief должен ссылаться на возможность развёртки в TOM, аналогичный Golden House full document.**

### Что NЕ должно быть в Brief после первого Discovery с Надеждой

- Точные имена менеджеров и распределение ролей — это уже Фаза 1.
- Списки конкретных поставщиков — это уже Фаза 1.
- Конкретные SKU — это уже Фаза 1.
- Финальные цифры цены — solution architect утвердит.

Discovery — это **карта местности**, не **полный план строительства**.

## 3. Discovery Completion Checklist

Используется агентом как self-check перед вызовом `generate_discovery_brief`. Основано на разделах 9-13 Golden House full document.

### 3.1. Покрытие требований клиента (раздел 9 Golden House)

Агент проверяет, что для каждого из 10 пунктов требований в Lead есть данные:

| # | Требование клиента | Какие Accounts Lead должны быть заполнены |
|---|---|---|
| 1 | Хранить заказы со всей цепочкой | `case_1_retail_flow`, `case_3_vendor_flow`, `case_6_logistics_flow`, `case_8_returns_flow` |
| 2 | Подзаказы с разными сроками | `case_3_vendor_flow` (упоминание подзаказов), `case_4_lead_time_flow` |
| 3 | Запрет корректировки после момента X | `case_7_closing_flow` (что подписывается) |
| 4 | Связать приход/оплату/отгрузку с заказом | покрыто кейсами 2, 5, 6 |
| 5 | Прайс розница/опт | `case_3_vendor_flow` (упоминание цен) |
| 6 | Роли (менеджер/продавец/кладовщик/закупки/логист) | `employees_count`, `outlets_count`, `internal_owner` |
| 7 | Статусы заказа по этапам | `case_1` ... `case_7` (упоминание состояний) |
| 8 | Фильтр заказов по поставщику | `must_keep[]` (явное упоминание) |
| 9 | Фильтр по стадии | `must_keep[]` или `owner_reports[]` |
| 10 | Стягивание в финрезультат | `case_7_closing_flow`, `owner_reports[]` (P&L) |

Если 9+ из 10 пунктов закрыты — фаза `BUSINESS_FLOW` считается завершённой.

### 3.2. Контрагенты и отделы (раздел 10 Golden House)

Агент должен зафиксировать, какие из 10 типов контрагентов есть у клиента:

| Сущность | Account Lead |
|---|---|
| Клиент (Customer) | implicit — есть у всех |
| Поставщик (Vendor) | `vendors_count` |
| Перевозчик (Carrier) | `logistics_providers[]` |
| Банк/эквайер (Bank) | `banks[]` |
| ПРРО (FiscalProvider) | `prro_provider` |
| Салон (Showroom) | `outlets_count` |
| Склад (Warehouse) | `warehouse_model` (есть/нет, транзит/постоянный) |
| Отдел закупок (Department:Procurement) | `has_procurement_department` |
| Отдел логистики (Department:Logistics) | `has_logistics_department` |
| Сотрудники с ролями (Employee) | `roles_used[]` |

### 3.3. Дашборды по ролям (раздел 11 Golden House)

Какие дашборды клиент хочет видеть. Спрашивается в Этапе 4 («Какие отчёты собственник смотрит?»):

| Дашборд | Account Lead |
|---|---|
| Операционный директор (воронка, кассовый поток) | `owner_reports[]` содержит соответствующие маркеры |
| Менеджер салона | `roles_used[]` содержит "manager" |
| Поставщики (рейтинг) | `must_keep[]` или `owner_reports[]` |
| Финансовый (P&L) | `owner_reports[]` содержит P&L |

### 3.4. Метрики автоматически доступные (раздел 12 Golden House)

Это даёт агенту аргументы в Этапе 5 («что вы получите из коробки»):

- Cycle time, время на каждом этапе
- Срок производства по поставщикам (фактический)
- Доля просрочек
- Личная маржа и комиссия менеджеров
- Сравнение между салонами
- P&L в реальном времени

Эти аргументы агент **достаёт из RAG**, чтобы конкретно показать клиенту value.

### 3.5. MVP-демо (раздел 13 Golden House)

Структура `create_prototype_stand` для Furniture Retail должна повторять эту структуру:
- Тестовые акторы Customer, Order, SubOrder, Vendor, Product, Employee.
- Кликабельный жизненный цикл одного заказа.
- Триггер на приближение готовности с реальным уведомлением.
- Граф связей.
- Дашборд «как у директора».
- Демо-сценарий «Заказ от Иванова на 50 000 ₴».

Если эти 6 элементов не присутствуют в Prototype — он не готов.

## 4. Quality Gates перед handoff

Агент перед вызовом `generate_discovery_brief` проходит self-check. Если хоть один gate не пройден — продолжает диалог или эскалирует.

### Gate 1 — Идентификация
Заполнены: `company_name`, `industry`, `outlets_count`, `employees_count`, `current_system`, `migration_trigger`, `target_deadline`.

### Gate 2 — Покрытие 8 кейсов
Описаны Accounts `case_1` через `case_8`. Каждый — хотя бы 1-2 предложения. Если хотя бы один кейс не известен (например, клиент не работает с возвратами) — `case_X_flow = "n/a"` (явное указание).

### Gate 3 — Покрытие требований ≥9/10
Из чек-листа 3.1 — закрыто 9+ из 10 пунктов.

### Gate 4 — Объёмы зафиксированы
Заполнены количественные Accounts фазы Volumes (хотя бы порядок величины).

### Gate 5 — Decision-making понятен
Заполнены: `internal_owner`, `decision_maker`, `budget_range` (диапазон), `success_criteria`.

### Gate 6 — Industry Pack matched
`industry_pack_match_pct >= 60` (иначе автоматическая эскалация на Custom-трек).

### Gate 7 — Нет активных триггеров эскалации
`escalation_signals[]` пуст или содержит только мелкие сигналы. Серьёзные — автоматически выводят в `ESCALATED`.

### Gate 8 — Roadmap graph approved (новое в v1.1)
В Этапе 5 DiscoveryAgent должен успешно вызвать `generate_roadmap_graph` и показать клиенту персональный план. Клиент должен подписать event `Roadmap.Approved` (валентность signer = product_owner). Без этого Brief не финализируется — клиент должен видеть и подтвердить свой план миграции.

Дополнительно:
- `forecasted_cutover_date` должен быть в пределах ±10% от `target_cutover_date` (иначе — пересогласовать сроки)
- Все Phase actors созданы со статусом `pending` или `ready` (для первой фазы Discovery — `completed`)
- Custom phases (если клиент упомянул специфику) добавлены в roadmap

**Если все 8 gate'ов пройдены — Brief генерируется, лид передаётся solution architect, Migration actor активен.**

**Если 6-7 пройдены — агент сам инициирует follow-up вопросы.**

**Если ≤5 — это сигнал, что Discovery провалился или клиент не подходит под текущий каталог Industry Packs. Эскалация на Custom-трек.**

## 5. Метрики качества Discovery-агента

Накопительные Accounts на акторе DiscoveryAgent для мониторинга качества:

### Метрики воронки
- `sessions_started`
- `sessions_completed` (прошли все 5 этапов)
- `sessions_dropped` (клиент перестал отвечать)
- `sessions_escalated` (триггер эскалации сработал)
- `completion_rate = sessions_completed / sessions_started`

### Метрики качества Brief
- `briefs_generated`
- `briefs_signed` (клиент пришёл на финальный звонок и согласовал)
- `briefs_rejected_by_sa` (solution architect отклонил Brief как недостаточный)
- `tom_developed` (Brief развёрнут в TOM уровня Golden House)
- `sign_rate = briefs_signed / briefs_generated`
- `tom_rate = tom_developed / briefs_signed`

### Метрики эффективности
- `avg_duration_minutes` (по проходящим до Brief)
- `avg_messages_to_brief`
- `time_to_first_message_response` (responsiveness)

### Метрики качества контента
- `human_overrides_count` (supervisor корректировал диалог)
- `false_promises_detected` (агент обещал то, чего нет) — нулевая допустимость
- `pricing_violations` (агент торговался) — нулевая допустимость

### Метрики A/B
Если параллельно работают `agent_v1` и `agent_v2`:
- разница `completion_rate`, `sign_rate`, `tom_rate`
- статистическая значимость на N=30+ сессиях

## 6. Что должно быть в supervisor-консоли

Со стороны Simulator должен быть человек, который видит активные диалоги Discovery-агента и может перехватить. Это **актор Supervisor** с дашбордом-проекцией на акторы Lead с фильтрами:

| Фильтр | Что показывает |
|---|---|
| `Lead.current_phase == X` | Активные диалоги на каждой фазе |
| `Lead.escalation_signals[] != []` | Лиды с накопленными красными флагами |
| `Lead.time_in_state > threshold` | Залипшие лиды (клиент перестал отвечать?) |
| `agent_v1 vs agent_v2 metrics` | A/B-сравнение |

Supervisor может:
- Открыть диалог в режиме просмотра
- Внести коррекцию в промпт (тогда событие `Override.Inserted` на DiscoveryAgent)
- Перевести лид на ручной режим (state Lead → `ESCALATED`)
- Дать обратную связь по диалогу (для тюнинга промпта)
